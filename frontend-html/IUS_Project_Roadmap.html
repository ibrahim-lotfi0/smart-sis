1. نظرة عامة والوضع الحالي للأنظمة (Status Board)

تهدف هذه الخطة إلى تركيز طاقات التطوير بنسبة 100% على بناء واجهات أمامية متكاملة لـ 3 بوابات رئيسية قبل البدء في مرحلة ربط قواعد البيانات الخلفية. هذا يضمن إخراج منصة خالية من الأخطاء وذات تجربة مستخدم مبهرة.

البوابة المستهدفة الصفحة / الجزء الحالة تفاصيل ما تم إنجازه بالكامل الخطوة القادمة (المتبقي في الواجهات)
بوابة الطالب لوحة التحكم (Dashboard) مكتمل إحصائيات كاملة، شريط جانبي متناسق، رسوم بيانية تفاعلية. ربط الحقول محلياً مع الذاكرة المؤقتة.
النتائج الأكاديمية (Results) مكتمل تأسيس GPA، تفصيل الدرجات، محاكاة طباعة السجل، ورسومات تقدم بيانية. ربط زر التحميل الفوري بصيغة PDF.
الواجبات والتسليمات مكتمل سحب وإفلات، تقويم مواعيد نهائية، بطاقات متباينة الأهمية. تخزين حالة التسليمات في الـ LocalStorage.
بوابة المشرف (Admin) لوحة الإدارة والرصد مكتمل الداشبورد الكلية، رصد الحضور لجميع الدفعات، تحليلات تفاعلية. بناء واجهة إدارة الطلاب والمواد (CRUD).
بوابة المحاضر لوحة الأستاذ الجامعي قيد التطوير تم تأسيس الهيكل العام للداشبورد والشريط الجانبي. تحديث التنسيق العام ليتوافق مع نظام "Royal Glass".
رصد درجات الطلاب متبقي - تصميم جدول يشبه الإكسل لإدخال درجات الطلاب.
دفتر الحضور والغياب متبقي - تصميم قائمة بأسماء الطلاب مع خانات اختيار تفاعلية للحضور.

توجيه هام للمرحلة الأولى

تم الانتهاء تماماً من ضبط اللمسات الجمالية والوظيفية لبوابة الطالب ونتائجها وجدولها وحضورها، وكذلك أدوات التحكم الخاصة بالمشرف. المرحلة الحالية تركز بالكامل على بناء بوابة المحاضر المتبقية كواجهات متناسقة.

2. توزيع المهام والأدوار البرمجية المقترحة

لإنجاز العمل بأعلى كفاءة وسرعة، نقترح توزيع العمل بين أفراد الفريق كالتالي (أو تطبيقها كمطور واحد في فترات زمنية متعاقبة):

1. مهندس الواجهات والتصميم (UI/UX)

  • تنسيق بوابة المحاضر لتتطابق مع الستايل الكحلي والذهبي الفاخر لباقي المنصة.
  • ضمان مرونة وتناسق الجداول والرسومات البيانية على أحجام شاشات الجوال.
  • إدراج تأثيرات الحركة المصغرة (hover effects & micro-animations).

2. مطور التفاعل والبيانات المحلية (JS Specialist)

  • بناء ملف `mockStore.js` ليعمل كـ "قاعدة بيانات محلية مؤقتة" في متصفح العميل (LocalStorage).
  • تأمين عملية تسجيل الدخول الوهمية وحفظ الـ Token وتوجيه المستخدمين حسب رتبهم.
  • تحديث رسومات Chart.js بمجرد تغيير درجات الطالب أو نسب حضوره من لوحة المشرف/الأستاذ.

3. مسؤول الجودة والتكامل والـ QA

  • تشغيل سكربتات التدقيق التلقائي للروابط لضمان خلو 100 صفحة من أي روابط مكسورة.
  • توحيد الهيدر والفوتر والـ favicon بشعار الكلية عبر كافة مستويات ومجلدات المنصة.
  • محاكاة سيناريوهات الاستخدام اليومي للتأكد من سلاسة وسرعة التحميل.

3. المسار الهندسي لتبادل البيانات في الواجهات (Mock Flow)

سيعتمد التطبيق بالكامل على محاكي بيانات موحد يعمل بالذاكرة المؤقتة. هذا المسار الذكي سيمكننا لاحقاً من ربط المنصة بأي قاعدة بيانات حقيقية (Supabase أو SQL Server) بمجرد استبدال دوال جلب البيانات المحلية بـ (Fetch API calls) فقط!

4. المراحل التنفيذية لإنهاء الواجهات بالكامل (Phased Plan)

خطة العمل الزمنية مقسمة لخطوات واضحة لتسليم بوابات (الطالب، المشرف، المحاضر) بأعلى جودة برمجية:

المرحلة الأولى: تهيئة وحفظ صفحات الطالب بالكامل (100% مكتملة)

تم بناء كامل صفحات النتائج، الجداول، رصد الحضور، الفيديوهات التعليمية، نظام الامتحانات التفاعلي، صفحات تسليم الواجبات وتوليد الـ Favicons بشكل موحد وقياسية.

المرحلة الثانية: بناء بوابة المحاضر وتنسيقها الملكي (قيد التطوير)

تصميم الواجهات الجمالية للوحة المحاضر. بناء شاشة رصد الدرجات الأكاديمية للمواد، وتأسيس صفحة الحضور اليومية التفاعلية مع تجهيز لوحة الإعلانات السريعة للطلاب.

المرحلة الثالثة: تطوير لوحة تحكم المشرف (CRUD) وحركات التفاعل

بناء واجهات المشرف الخاصة بإضافة وتعديل وحذف الطلاب والمواد (CRUD Views)، وربطها بشكل مباشر بالـ LocalStorage لتعمل بكفاءة وحيوية تامة أمام لجان التحكيم.

المرحلة الرابعة: المراجعة الكلية والتسليم والانتقال للباك اند

إجراء تدقيق كامل لكافة الروابط والصور، تسليم واجهة النظام بالكامل للكلية كـ (Premium Web Prototype)، ثم وضع خطة ربط الخوادم (Supabase / SQL Server API Server) وبث البيانات الحية وتطبيق الجوال.

نقطة التسليم والجاهزية

باتباع هذا المسار الدقيق للواجهات أولاً، نضمن أن يكون لدينا منتج نهائي رائع وخالٍ تماماً من المشاكل التصميمية، مما يسهل كثيراً عملية التكامل مع تطبيق الفلاتر والباك اند لاحقاً دون الحاجة لتعديل التصاميم مجدداً.
